**Codename:** sunny-anchor
**Run date:** 2026-05-18
**Driver:** Claude Code, acting as the user. Real MiniMax-M2.5. Real Electron build (`oscar-gc` 206 MB, Sprint 6 cut). Xvfb on `:99`. Two cold-start sessions back-to-back, profile reset between.
**Sprint 6 artefact under test:** agent-driven first-launch onboarding (ADRs 008–014).

This is a user-research report, not a build report. Sprint 7 produced no product code changes. The dogfood driver (`scripts/dogfood/dogfood.sh` + `ui/desktop/scripts/dogfood-driver.mjs`) is test harness only.

## 2. Methodology

- Pre-flight: deleted `~/.config/oscar/profile.json`; verified MiniMax round-trip with `goose run --no-session`; verified `oscar-onboarding-mcp` built and desktop binary present.
- Launched the packaged Oscar GC binary under Xvfb with CDP debug port 9223.
- Drove the chat via Playwright/CDP: fill `.oscar__chat-input` → click `.oscar__chat-send` → poll for response stabilisation. Each persona turn computed at runtime by CC, not pre-scripted.
- After every agent turn: screenshot to `screenshots/<session>/NN-label.png`; turn appended to `/tmp/oscar-dogfood/turns.json` as DOM-side backup.
- After both sessions: canonical transcripts extracted from `~/.local/share/goose/sessions/sessions.db` (including tool calls and the LLM-emitted closing message that the UI did not render).
- Profile verification: `cat ~/.config/oscar/profile.json` compared field-by-field against the conversation.
- Cold-relaunch verification: app quit, relaunched, confirmed onboarding did not retrigger and the sidebar persisted.

A small deviation from the plan: the Node driver lives at `ui/desktop/scripts/dogfood-driver.mjs` rather than `scripts/dogfood/run-onboarding-session.mjs` because Playwright is installed at `ui/node_modules/` and ESM resolution requires the importing file to live underneath. The user-facing entry point `scripts/dogfood/dogfood.sh` still sits at the documented path.

## 3. Headline findings

1. **One P0 product bug, reproducible 2/2 runs.** The agent's closing message ("Welcome to Oscar GC. Your practice areas are listed in the sidebar — pick one to begin.") is generated by the LLM and persisted to the session DB, but the user never sees it. The `OscarOnboardingGuard` polls `profile.json` every 1500 ms; once the tool call writes the file, the guard unmounts the onboarding view before the LLM's next response streams in. The user is dropped into the Hub with no acknowledgment that the save succeeded and no bridge to the sidebar. The intent of ADR-014 is partially violated.
2. **Profile capture itself is excellent.** Both sessions wrote JSON that matches the conversation precisely. Canonical role-slug derivation works (`counsel` for "Commercial Counsel", `other` for "lawyer"). Null-handling for declined fields is clean. Default `body` text is preserved verbatim from the seed. Custom `body` authoring for user-added areas is concise and parallels the seed pattern. Size-band natural-language mapping ("around 450" → `201-1000`) works.
3. **Agent voice is mostly on-brand but inconsistent.** The system prompt's voice ("professional, direct, peer of a lawyer; no chatbot tics") lands cleanly most of the time, but small tics ("thanks", "got it", "all 13 it is", "when you say the word, I'll save it") leak through. The system prompt's explicit forbid list ("Hey!", "Great!", "Awesome!") doesn't cover these adjacent forms.
4. **The required recap is non-deterministic.** Primary session: agent skipped the recap until pushed. Edge session: agent gave the recap unprompted. System prompt requires it as part of P4; LLM compliance is unreliable. This is a P1 because a user expecting verification may not get it.
5. **Hub landing is generic and impersonal.** Post-onboarding, the user lands on the same Oscar GC marketing page they would have seen pre-onboarding. There is no "Welcome, Daniel" line, no acknowledgment of what was just configured. Combined with finding (1), the user is left wondering whether anything happened.

## 4. Friction log

### P0

**P0-A — Closing message never rendered to the user (race condition).**
- **Turn:** primary 13:41:34, edge 13:48:48 — final assistant turn in `sessions.db`.
- **What happened:** Agent calls `finalize_profile`. Tool succeeds. `~/.config/oscar/profile.json` is written. Before the LLM's next streaming response (the closing message) reaches the React DOM, `OscarOnboardingGuard` polls (≤1500 ms after profile write), sees the file exists, unmounts `OscarOnboardingView`, mounts the Hub. The closing message is in `sessions.db` but no React turn was ever rendered for it.
- **Why it hurts:** The user finishes a five-minute interview, types "save it", and instantly sees a different screen — with no confirmation that the save succeeded, no bridge to the sidebar, no name on screen. A real lawyer's first reaction is "did that actually work?" Worse, the bridge that tells them to use the sidebar is missing, so they may not realise the sidebar entries are clickable.
- **Reproducible:** 2/2.
- **Fix direction:** Three viable options for Sprint 8 to decide between, then write an ADR for the chosen path:
  - (a) Agent emits closing message *before* the tool call (changes the agent contract — closing line becomes a setup line; ambiguous if tool fails).
  - (b) Guard waits for an explicit session-complete signal, not just profile-file existence (richer polling state).
  - (c) Move the welcome bridge off the chat surface entirely — render it as a one-time Hub banner that reads the just-written profile. (Recommended — also addresses finding 5 and P1-D below.)

### P1

**P1-A — Recap delivery is non-deterministic.**
- **Turn:** primary 13:40:32 vs edge 13:48:12.
- **What happened:** Primary skipped the recap until the user pushed back. Edge delivered the recap unprompted. The system prompt's P4 spec says recap is required; LLM compliance is inconsistent.
- **Why it hurts:** A user expecting verification before save may not get it (primary case). A user who didn't ask for it may be surprised by an extra paragraph (edge case). Either way the contract is unstable.
- **Fix direction:** Move the recap off the LLM entirely — render a deterministic profile preview in the chat surface between the agent's "ready to save?" and the user's final confirmation. Falls out naturally from fix (c) above.

**P1-B — Two contradictory questions in same turn (primary P3 end).**
- **Turn:** primary 13:40:09 — "Looks close to your practice, or want to drop or add anything? ... What do you want to keep?"
- **What happened:** Agent asked both the drop-mode framing (per spec) and the inverse keep-mode framing in the same turn.
- **Why it hurts:** Two opposing framings make the user choose which to answer. Daniel (detail-oriented) ignored the "keep" framing; a less confident user might enumerate inclusions instead of exclusions and end up with a different profile.
- **Edge session did not reproduce:** asked only the drop framing.
- **Fix direction:** Tighten the system prompt's example sentence. Show the *exact* question the agent should ask, not a paraphrase.

**P1-C — Recap factual condensation: dropped a detail (primary).**
- **Turn:** primary 13:40:59 — recap says "drives, controls, switchgear" but user said "drives, controls, switchgear, cabling".
- **Why it hurts:** A commercial counsel notices when details get dropped. Erodes trust in the recap.
- **Fix direction:** Same as P1-A — move the recap off the LLM so it can't editorialise.

**P1-D — Hub landing has no acknowledgment of just-completed onboarding.**
- **Turn:** post-tool-call in both sessions.
- **What happened:** Hub renders the same generic "Oscar GC — An in-house legal agent platform" hero copy in both pre-onboarding state (theoretical — never seen, blocked by guard) and post-onboarding state (actually shown). The hero copy doesn't change. No welcome-by-name, no "your sidebar has been populated", no entry point.
- **Why it hurts:** Combined with P0-A, the user is parachuted from a five-minute conversation into a marketing page. The sidebar is the only visible delta and it's narrow and quiet.
- **Fix direction:** First-arrival Hub banner reading from `profile.json` (e.g., "Welcome, Daniel — your sidebar reflects the practice areas you cover. Pick one to begin.") Dismissible. One-time. Could also serve as the closing-message bridge per P0-A.

### P2

**P2-A — Chatbot tics outside the explicit forbid list.**
- Tics observed: "thanks" (primary 13:39:24), "got it" (primary 13:39:41, edge 13:47:11, 13:47:41, 13:47:56), "all 13 it is" (edge 13:48:12), "when you say the word, I'll save it" (primary 13:40:59), "we wrap" (edge 13:48:30).
- System prompt forbids only "Hey!", "Great!", "Awesome!". These are adjacent.
- Fix direction: extend the forbid list, or shift the rule to a positive framing ("respond like a senior associate, not a customer-success agent").

**P2-B — Industry stored as long natural-language string.**
- `corporate.industry`: `"Industrial electrical and automation components distribution"`. Valid per schema (string), but not normalised — no canonical industry token, no length cap.
- Why it might hurt later: if the industry field ever drives template selection or counterparty defaults, sentence-fragment values are awkward to switch on.
- Fix direction: optional canonical industry slug field, or a max-length on `industry`, or both.

**P2-C — MiniMax explanation in primary was slightly over-narrated.**
- Turn: primary 13:40:59 — "MiniMax is an LLM provider — the model is MiniMax-M2.5. It's the engine Oscar runs on for reasoning and drafting."
- System prompt says don't narrate internal mechanics. But the user asked directly, so some response is required. The response is acceptable but adjacent to the rule.
- Edge handled the same probe correctly with "MiniMax-M2.5 is set on the system — confirm and we wrap."
- Fix direction: explicit guidance in the system prompt for handling direct user questions about provider/model.

**P2-D — Agent re-asked the name question after deflection (edge).**
- Turn: edge 13:46:50 — "Why do you need my name?" → "So the sidebar shows who you are. What should I call you?"
- System prompt says "Do not re-ask the same question" after objections. But "why?" is a clarification request, not an objection — the system prompt's intent is ambiguous here.
- Fix direction: clarify the rule. Re-asking with a one-clause "why" after a clarification request is fine; re-asking after a "no" is not.

**P2-E — Auto-generated session names from session manager bleed product semantics.**
- Edge session got auto-named "Anonymous legal inquiry" by goosed's session-naming heuristic. If a session-history surface ever displays this, it's an awkward label for what is conceptually "the user's onboarding."
- Out of strict scope for onboarding, but worth flagging because the onboarding flow currently creates a session with a name the user never sees but that future surfaces might.

**P2-F — Session DB carries unrelated test-run pollution.**
- `sessions.db` contains nine prior sessions from Sprint 6 development (CLI tests, test profiles, etc.). If any future Oscar surface lists recent sessions, these will appear.
- Out of strict scope. Tracked here so it isn't forgotten.

## 8. Recommendations for Sprint 8+

In priority order:

1. **Fix P0-A (closing message race).** Pick between options (a)/(b)/(c) in the friction log; (c) is recommended because it also closes P1-A, P1-C, P1-D. Write an ADR (proposed: ADR-015 "Welcome bridge moves from agent turn to Hub banner") at decision time, ahead of the change.
2. **Fix P1-D (Hub landing acknowledgment).** Naturally bundled with (1) if you go with option (c). One-time first-arrival welcome banner reading from `profile.json`.
3. **Fix P1-B (two-questions-in-one-turn).** Tighten the system-prompt P3 phrasing — show the exact question text, not a paraphrase example.
4. **Defer P1-A and P1-C** to whatever the (1) fix shape is — if recap moves to client-side rendering, the LLM inconsistency goes away by construction.
5. **Sweep P2-A (extended forbid list)** when next touching the system prompt.
6. **Defer P2-B, P2-C, P2-D** to future onboarding-related sprints.
7. **Park P2-E, P2-F** for the post-onboarding flow work (Sprint 8+'s scope).

**Pattern observation worth establishing now:** Sprint 7 validated the "CC as the user" dogfood pattern. Five frictions in this report (P0-A, P1-A, P1-B, P1-C, P1-D) would not have been caught by Sprint 6's unit tests, bundle greps, or state screenshots. P0-A in particular requires running a full conversation against a real provider — no synthetic test would have hit the race condition because the timing depends on real LLM streaming latency vs. real guard poll cadence. Future UX-heavy sprints should bake this pattern in as part of sprint-close, before declaring done.

The dogfood driver (`scripts/dogfood/dogfood.sh` + `ui/desktop/scripts/dogfood-driver.mjs`) is documented in `RUNBOOK.md` under "Dogfood capture" and is reusable for future UX sprints.
