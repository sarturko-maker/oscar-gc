# ADR-060 — Integrations trust tiering + community-tier consent gate

Status: accepted
Date: 2026-05-19
Sprint: 17

## Context

The Integrations registry (ADR-059) catalogues a mix of:

- **Oscar-bundled MCPs** (`oscar-fs`, `oscar-memory`, `oscar-onboarding`, `redline`): authored / vendored by us; license-clear (Apache 2.0 / MIT); audited zero-egress at build time per ADR-042; always-on or recipe-declared by us.
- **Trusted-by-design MCPs** (CourtListener): authored by third-party non-profit (Free Law Project), free public data, no subscription gate, no proprietary data flow.
- **Community MCPs** (Ironclad, DocuSign, Slack, Google Drive, Definely, Everlaw, etc.): authored / hosted by commercial vendors. Adding one widens the agent's outbound network surface (ADR-042 audit floor); some require paid subscriptions; data leaves the device under the SaaS's TOS.

Different tiers warrant different UX:

- **Bundled** belongs in the catalog for *transparency* (every MCP a lawyer might use is visible — brief's load-bearing principle), but the user shouldn't "Add" a thing that's always-on.
- **Trusted** can install with a one-line confirmation (no surprise, low stakes).
- **Community** requires explicit informed consent — license posture, subscription posture, hostname, maintainer, egress widening — because the user is the only safeguard until bubblewrap-class sandboxing lands (ADR-042 / Sprint 15+ deferral).

## Decision

Three tiers (`SecurityTier = 'bundled' | 'trusted' | 'community'` per overlay entry — ADR-059), each with a different Add affordance:

- **Bundled**: card renders an `Always-on` badge in place of the Add button. No click affordance. Card is in the list for transparency only.
- **Trusted**: Add button → short confirmation modal:
  ```
  Add CourtListener to IP?
  This is a trusted integration.
  - Talks to: mcp.courtlistener.com
  - License: apache-2.0 (wrapper)
  - Maintained by: Free Law Project (non-profit)
  [Cancel] [Add]
  ```
- **Community**: Add button → full trust prompt:
  ```
  Add Ironclad to Commercial?
  This is a community-tier integration.
  - Talks to: mcp.na1.ironcladapp.com (over HTTPS).
  - License: proprietary (vendor-managed).
  - Subscription: requires a paid Ironclad subscription.
    → You will be prompted by Ironclad on first tool call.
  - Maintained by: Ironclad, not Oscar GC.
  - Data leaving your device: queries you ask the agent
    that the agent decides to forward to Ironclad, and
    any responses.

  Oscar GC does not sandbox network egress today. Adding
  this integration widens where your agent talks. Real
  sandboxing is planned in a later release.
  [Cancel] [Add and acknowledge]
  ```

**Consent record**: on "Add" / "Add and acknowledge", `trust_acknowledged: true` is written into the entry in `installed_integrations.json` (ADR-061). On Cancel: no write. **No "skip the trust prompt" path** — the only way past the prompt is explicit consent. The button label encodes the action: trusted uses "Add"; community uses "Add and acknowledge".

## Rationale

- **Honest transparency over hidden curation.** The brief: "every MCP a lawyer might use is *visible*, labelled honestly". Bundled-as-card preserves transparency without misleading the lawyer into thinking they need to install it.
- **Tier-shaped friction is honest.** Trust prompts that fire on every Add (bundled too) would condition the user to click through; tier-shaped friction reserves the heavy prompt for the case that warrants it.
- **Consent is the security backstop until sandboxing lands** (ADR-042 deferral). The trust prompt is the structural floor; consent record is the durable audit. Real sandboxing (bubblewrap / network namespaces) becomes load-bearing in Sprint 15+ and supersedes this trade-off.
- **Subscription posture is non-negotiable in copy.** "You will be prompted by Ironclad on first tool call" — Oscar GC doesn't pretend to provide what the SaaS provides; the trust prompt makes the gap visible at install time, not at first failure.
- **Maintenance attribution is named.** "Maintained by Ironclad, not Oscar GC" makes the privity boundary explicit — when the wrapper breaks or changes, the user knows whose support channel to use.

## Consequences

- `IntegrationCard.tsx` renders one of three states per tier. `ConfirmAddModal.tsx` branches on `security_tier`. Both shipped in P3.
- Adding a new entry to `INTEGRATIONS_OVERLAY` requires picking a tier — fail-closed if missing (registry loader rejects entries without `security_tier`).
- Trust-prompt copy is hand-written per entry where the hostname / maintainer detail differs; the *structure* is enforced by the modal component, the *strings* come from the overlay's `service_endpoint_host` and `facts_note`. No prompt is autogenerated from generic templates.
- Re-tiering an entry (e.g., promoting community → trusted after a vendor audit) is a one-line overlay edit. Future ADR records the audit conclusion.
