# ADR-015 — Welcome bridge moves from agent chat turn to a one-time Hub banner

Status: accepted
Date: 2026-05-18
Sprint: 8

## Context

Sprint 7's dogfood pass (2/2 runs against real MiniMax-M2.5, `docs/dogfood/sprint-7/README.md` §4 P0-A) surfaced a render-timing bug. The agent's closing message ("Welcome to Oscar GC. Your practice areas are listed in the sidebar — pick one to begin.") is generated by the LLM and persisted to `~/.local/share/goose/sessions/sessions.db`, but the user never sees it. `OscarOnboardingGuard` polls `~/.config/oscar/profile.json` every 1500 ms; when `finalize_profile` writes the file the guard unmounts `OscarOnboardingView` before the LLM's next streamed turn lands in the DOM. The user is dropped onto a blank Hub with no acknowledgment that the save succeeded and no bridge to the sidebar.

The conversation itself is good (Sprint 7's verdict: every captured profile was correct, the system prompt holds, the recipe surface is sound). The bug is post-conversation rendering.

Three fix options were evaluated in Sprint 7's report:

1. **Agent emits closing message *before* the tool call.** Rejected — changes the agent contract; if the tool fails the closing line is already on screen.
2. **Guard waits for an explicit session-complete signal, not profile-file existence.** Rejected — richer polling state, but addresses neither P1-D (Hub still impersonal post-onboarding) nor P1-A/C (LLM-recap non-determinism).
3. **Move the welcome bridge off the chat surface; render it as a one-time, dismissable Hub banner that reads `profile.json`.** Recommended — also closes P1-A, P1-C, P1-D by relocating the verification surface from the LLM's recap to a deterministic banner.

## Decision

Option (3). The post-onboarding welcome bridge is a one-time, dismissable banner rendered on Hub. The banner reads `~/.config/oscar/profile.json` at mount; uses `user.name` for personalisation (`Welcome to Oscar GC, {firstName}.` or `Welcome to Oscar GC.` if declined); body text mirrors the system-prompt closing message. Dismissal persists via `localStorage` key `oscar.hubWelcomeDismissed`.

## What is moving / not moving

**Moving:** the rendering location of the welcome bridge — from `OscarOnboardingView`'s message list to `Hub`. The agent still emits its closing message per ADR-010 §P4; that message still persists to `sessions.db`; it just stops being load-bearing for what the user sees on screen.

**Not moving:** agent persona, system prompt, conversation contour, or tool surface (ADRs 008/009/010/013 unchanged). Profile schema (ADR-011 unchanged). Post-onboarding entry point (ADR-014 unchanged — Hub remains the landing).

## Rationale

- Render-deterministic. The banner reads `profile.json` after the guard mounts the Hub; no streaming-timing race possible.
- Personalisation closes P1-D — the Hub no longer looks identical pre- and post-onboarding.
- Reading `profile.json` (not `sessions.db`) deprecates the LLM recap as the verification surface — P1-A (non-determinism) and P1-C (detail drop) become cosmetic, not load-bearing.
- `localStorage` over a `profile.json` field — no schema change, no sibling-MCP-repo edit, no new IPC channel.
- Reuse evaluation: `AlertBox` (`components/alerts/AlertBox.tsx`) and `GroupedExtensionLoadingToast` are Goose's existing notification primitives. Both are alert/toast-styled (11px text, color-coded by severity, top-right corner). Wrong register for a one-time editorial welcome card. Declined — Sprint 8 introduces the first banner in the `.oscar__*` namespace, ~40 LoC + ~25 LoC CSS.

## Consequences

- ADR-014's deferred "additive change to `Hub.tsx`" is activated this sprint (ADR-014 §Rationale, last bullet). ADR-015 extends ADR-014; does not supersede it.
- Known limitation: profile re-onboarding doesn't re-show the banner. If a user deletes `profile.json` and re-runs onboarding, the `oscar.hubWelcomeDismissed` flag persists. Acceptable for v1; re-runs are manual/rare. Future fix (tie dismissal to `profile.completed_at`) only if user-reported.
- Banner copy is fixed-and-deterministic for v1. Variation by role/company is a future ADR if surfaced.

## Supersedes

None. Extends ADR-014.
