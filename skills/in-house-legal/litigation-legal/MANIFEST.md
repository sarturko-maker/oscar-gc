## CLAUDE.md

The plugin CLAUDE.md is a TEMPLATE for the per-user practice profile, written by cold-start. It has three role branches (`in-house`, `firm-associate`, `solo`) and matching sections (board memo + reserve + settlement-authority ladder for in-house; case theory + pivot fact + seed brief for firm-associate; caseload + contingency + retainer for solo). For Oscar GC we strip the firm-associate and solo branches under ADR-033, replace `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md` with Oscar's profile + per-area memory paths, and drop the cache-migration logic (no upstream cache to migrate from). The in-house posture sections (risk calibration, materiality / ASC 450, settlement-authority ladder, outside counsel bench, insurance profile, board / audit committee memo style) all stay — they ARE the in-house frame.

## In-house vs outside-counsel posture

Litigation-legal is the plugin most exposed to the outside-counsel-vs-in-house tension flagged in the brief. The plugin's framing is unambiguously **in-house portfolio counsel managing matters that outside firms execute** — the README opens with "Built for counsel who own many matters at once, most of which are run by outside firms" and the agents/commands (matter-intake, portfolio-status, oc-status, legal-hold portfolio rollup, docket-watcher) are dashboard / orchestration tools an in-house lawyer uses to direct outside counsel work. Most skills are clean keeps under that frame.

The borderlines are **brief-section-drafter** and **deposition-prep**: these produce work product that outside counsel typically owns in a litigated matter. Both are kept-borderline because (a) in-house counsel use them to draft sections for OC review, pressure-test OC's drafts, or own depo prep for regulatory enforcement matters they handle directly; and (b) the skills' guardrails (Rule 11 framing, citation hygiene, the non-lawyer gate, the candor-about-weak-arguments rule) are exactly the rails an in-house user needs when they DO draft. Skills that produce outside-counsel-only work product (complaints, motion briefs in full) are NOT in this plugin — claim-chart is the closest, and it's framed for in-house pressure-testing of OC charts ("the killer outputs are (a) a chart that says what evidence goes with what element and (b) a gap list that tells the attorney what's missing"). Flag for orchestrator: confirm whether Oscar's MVP scope for disputes areas includes brief-drafting workflows or stops at the portfolio / triage / hold / chronology layer; if the latter, brief-section-drafter and deposition-prep can move to a later sprint.

The cold-start-interview is dropped from the bundle (cold-start happens at the Oscar onboarding-MCP level). Customize and matter-workspace are dropped — Oscar's profile editor and one-client / per-element-scope memory replace them.

## Cold-start interview extraction

The plugin's cold-start asks role (in-house / firm-associate / solo) and side (plaintiff / defense) before any substantive pillar. For Oscar GC the role is fixed (in-house) and side defaults to defense (with per-matter overrides). The two priority-1 onboarding questions extracted below capture the in-house disputes-spine essentials that EVERY downstream skill reads: (1) the matter portfolio's volume and risk shape, which feeds severity bands, materiality thresholds, and triage calibration; (2) the outside-counsel engagement model, which feeds oc-status cadence, budget posture, and the directive style every OC communication uses. Questions are phrased generally enough to work across commercial-disputes, employment-disputes, ip-disputes, and regulatory-disputes — each area's onboarding flow will layer its own area-specific questions on top.

## Output Types proposed for this plugin

Disputes-spine outputs are written by the active disputes area, not by litigation-legal itself. The active practice-area context (commercial-disputes / employment-disputes / ip-disputes / regulatory-disputes) supplies the `<Practice Area Name>` segment of the output path. All paths take the form `~/Documents/Oscar/<Practice Area>/<Output Type>/`:

- **Demand Letters** — written by demand-draft (paired with demand-intake)
- **Brief Sections** — written by brief-section-drafter
- **Claim Charts** — written by claim-chart (patent or civil element)
- **Chronologies** — written by chronology
- **Deposition Prep Memos** — written by deposition-prep
- **Legal Hold Notices** — written by legal-hold (issue / refresh / release)
- **Privilege Log Reviews** — written by privilege-log-review
- **Matter Status Reports** — written by matter-briefing (one matter, deep) and matter-update (event log)
- **OC Status Reports** — written by oc-status (weekly portfolio sweep, one report per matter + summary)
- **Portfolio Status Reports** — written by portfolio-status
- **Inbound Triage** — written by demand-received and subpoena-triage (cross-referenced because both triage inbound paper and route to matter creation)

## Borderline / open questions for orchestrator

1. **Brief-section-drafter and deposition-prep scope.** Confirm whether MVP disputes-area workflows include brief drafting and depo prep, or stop at the triage / chronology / hold / portfolio layer. If the latter, defer these two to a later sprint.
2. **Docket-watcher dependencies.** The agent requires Trellis + CourtListener MCPs; neither is in Oscar's MVP integration set. Confirm whether to ship the agent with a "no research connector — degrade gracefully" path (the SKILL.md already supports this for citations), or defer.
3. **Cross-area matter store.** The matter ledger (`_log.yaml`) and per-matter files (`matter.md`, `history.md`, `chronology.md`, `legal-hold-v[N].docx`) need a per-disputes-area home in Oscar's memory MCP. A single matter typically lives in one disputes area, but cross-area links (an employment matter that triggers an IP claim, a commercial dispute that surfaces a regulator inquiry) are common — confirm whether matters are scoped to one area with `related_matters` cross-area pointers, or live in a shared litigation root with an area tag.
4. **Conflicts gate.** Every matter-touching skill has an unbypassable conflicts gate that reads `_log.yaml`. Oscar's memory MCP needs to expose an equivalent read so the gate still bites — confirm shape.
5. **In-house counsel privilege jurisdiction note (priv-log review).** The skill flags that EU competition (Akzo Nobel), German Syndikusanwalt, and France/Belgium regimes change in-house privilege. Oscar's company-profile jurisdiction footprint feeds this — confirm the path from profile to skill is wired.
6. **Cold-start interview replacement scope.** The plugin's cold-start writes 3 pillars (risk calibration, landscape, house style) over 10-15 minutes. Oscar's area-level onboarding will need to cover the same ground per disputes area. Confirm whether each area's onboarding adds the litigation-spine pillars on top of area-specific questions, or whether a one-time shared "disputes spine" onboarding runs once across all four areas.
